안녕하세요. 오늘 주제는 한 문장입니다. "코드는 비공개로 지키고, 로직은 공공재로 푼다." 지금까지 우리 코드 안의 로직을 이해하려면 결국 코드에 접근하거나, 그 코드를 아는 사람한테 물어야 했어요. 이 봇은 그 둘을 분리합니다. 코드는 사내 자산으로 그대로 가두고, 그 안의 로직은 팀 누구나 꺼내 쓸 수 있는 공공재로 바꾼 이야기를 해보겠습니다.
01 문제

로직이 코드 속에 갇혀 있다

로직을 이해하려면 코드에 접근해야 하고, 결국 그 코드를 아는 한 사람에게 묶입니다. 코드 접근 권한이 곧 로직 이해 권한이 되는, 지식 비대칭 구조죠.

해법의 핵심은 분리입니다. 지금까지 코드 접근과 로직 이해가 한 덩어리로 묶여 있었다면, 이걸 떼어냅니다. 코드 자체는 보안 경계 안에 비공개로 가두고요. 그 안의 로직, 즉 '어떻게 동작하는가'라는 이해는 누구나 꺼내 쓰는 공공재로 풉니다. 그리고 그 둘 사이에서 코드를 안전하게 읽어 이해만 내보내는 번역기 역할을 봇이 합니다.

비공개 코드는 경계 안에서만 읽고, 밖으로 나가는 건 코드가 아니라 이해(로직)뿐. 그 경계를 Hermes가 만듭니다.

구조로 보면 이렇습니다. 누구나 슬랙으로 질문을 던지면, Hermes가 관문 역할을 하면서 경계 안에서만 코드를 읽습니다. 모노레포는 read-only로 경계 안에 머물고, 밖으로 나가는 건 코드 원문이 아니라 '이렇게 동작한다'는 이해뿐이에요. 즉 Hermes가 비공개와 공공화를 가르는 경계선 그 자체입니다. 이제 이 장치를 깊게 보겠습니다.
HERMES 심화 ① — 정체

Hermes Agent란?

NousResearch의 오픈소스 AI 에이전트 프레임워크(MIT). 모델이 아니라, 모델을 ‘일하는 에이전트’로 만들어주는 몸통. 캐치프레이즈는 “The agent that grows with you.”

🔌
why 1

메시징 15종+ 내장

Slack·Telegram·Discord·Email… 채널 붙이는 코드가 필요 없음

🛠
why 2

도구 + MCP

파일 읽기·터미널 실행이 기본 + MCP 지원 → 우리 목표에 즉시 부합

🧠
why 3

스스로 학습

맥락을 정리해 skill을 자동 생성·진화시킴

Hermes는 오픈소스 LLM Hermes 시리즈를 만든 NousResearch의 에이전트 프레임워크입니다. 제가 고른 이유 셋: 메시징 연동 15종+ 기본 내장, 파일·터미널 도구 + MCP 기본 제공, 그리고 질문마다 맥락을 정리해 스스로 skill을 만들어 진화한다는 점. 이 세 번째가 결정적 차별점이라 뒤에서 따로 봅니다.
HERMES 심화 ② — 구조

3계층 아키텍처

Layer 1
Gateway

메시징 어댑터

슬랙·텔레그램 등 들어온 메시지를 공통 포맷으로 정규화 · 세션/인증/cron 처리 → 슬랙 연동이 설정 몇 줄로 끝

Layer 2
AIAgent

오케스트레이션 엔진

모델(provider) 선택 · 시스템 프롬프트 조립(SOUL/메모리/skill) · 도구 실행 · 재시도 · 컨텍스트 압축

Layer 3
Backend

실제 실행 위치

Local / Docker / SSH / Daytona / Modal… → 우리는 Docker로 격리

Hermes는 3계층입니다. Gateway는 메시징 담당이라 슬랙 연동이 설정 몇 줄로 끝나고, AIAgent는 두뇌라기보다 오케스트레이터로 모델 선택·프롬프트 조립·도구 실행·압축을 맡습니다. Execution Backend는 코드를 실제로 어디서 돌릴지인데 우리는 보안 때문에 Docker를 골랐어요. 셋이 분리돼 있어 두뇌는 두고 실행 환경만 안전하게 바꿀 수 있습니다.

모델 변천사 : provider 한 줄 교체

  • 코드 이해·설명 품질 우수 → 첫 선택
  • ⚠ 모노레포 토큰 폭증 → rate limit + 비용

CodexNOW

  • 설정 한 줄로 provider 교체
  • 코드 변경 0으로 전환 완료
3

Gemini

  • 큰 context + Workspace 비용 이점
  • 모노레포 적합성 테스트 예정

‘일단 붙여보고 아니면 갈아탄다’를 부담 없이 — provider-agnostic = 실험 비용 거의 0.

모델을 두 번 갈아탔습니다. 처음 Claude는 품질이 좋았지만 모노레포 특성상 토큰이 빠르게 쌓여 rate limit과 비용이 부담이었어요. 지금은 Codex인데, Hermes가 provider를 설정 한 줄로 바꾸는 구조라 코드 한 줄 안 고치고 전환했습니다. 다음엔 Gemini도 비교할 거고요. 이렇게 부담 없이 실험할 수 있는 게 provider-agnostic의 진짜 가치였습니다.
HERMES 심화 ⑤ — 🔒 비공개 경계

코드는 비공개로 지킨다

  • 🐳Docker 컨테이너 실행 — 맥북에서 컨테이너로 구동해 호스트와 분리
  • 📁모노레포 읽기 전용 마운트(:ro) — 코드 수정·반출이 구조적으로 불가능
  • 🧱Hermes 기본 안전장치 — 위험 명령 승인(command approval) · 권한 드롭 · LLM에 비밀키 노출 차단(credential stripping) · 세션 격리

코드는 경계를 넘지 못합니다. 밖으로 나가는 건 코드가 아니라 ‘이해’뿐 — 이게 ‘공공화’가 ‘유출’이 되지 않게 하는 선입니다.

공공화의 전제는 비공개가 확실히 지켜진다는 거예요. 그래서 경계를 분명히 했습니다. 맥북에서 Docker로 돌리고, 모노레포를 읽기 전용으로 마운트해서 코드 수정도 반출도 구조적으로 불가능합니다. Hermes 자체도 위험 명령 승인, 권한 드롭, 비밀키 차단, 세션 격리가 기본이고요. 즉 코드는 경계를 못 넘고, 밖으로 나가는 건 '이해'뿐입니다. 이 선이 있어야 로직을 푸는 게 코드를 푸는 게 아니게 됩니다.
HERMES 심화 ⑥ — 도입

어디에 맞고, 어떻게 시작하나

🎯 언제 맞나

  • 정책·승인 절차가 필요하거나 목표가 명확한 반복 업무
  • ‘이 사람은 주로 이런 일을 한다’를 학습·진화 → 사무·운영 자동화에 특히 적합
  • 특정 도메인 질문에 일관된 형식으로 답하기 (= 우리 봇)

⚡ 도입 난이도

  • 한 줄 설치 스크립트 → 약 15분이면 첫 대화
  • hermes gateway setup 으로 슬랙 연동
  • terminal.backend=docker 한 줄로 격리
  • GUI 대시보드 제공 · 최소 요건 context 64k↑ (Claude·GPT·Gemini 충족)

인프라를 새로 짜는 게 아니라 설정으로 조립하는 도입 → 팀 내 다른 반복 업무로 확장 가능.

그래서 '우리가 또 어디에 쓸까'를 정리했어요. Hermes는 정책·승인이 있거나 목표가 분명한 반복 업무, 학습·진화가 중요한 사무·운영 자동화에 잘 맞습니다. 도입 난이도도 낮아요. 한 줄 설치로 15분이면 첫 대화, 슬랙 연동·Docker 격리는 각각 한 줄, GUI도 있고 모델은 64k 이상이면 됩니다. 인프라가 아니라 설정으로 조립하는 도입이라 확장도 쉽습니다.

## 주의 / 사이드이펙트
- 관련 영향 범위 · 주의사항
  • 📍 파일 경로 + 라인 번호로 바로 확인 가능
